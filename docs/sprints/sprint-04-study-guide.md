## 3. Key Concepts Explained

### What is Text-to-Speech (TTS)?

Text-to-speech converts written text into spoken audio. The software analyzes the text, applies linguistic rules about pronunciation and rhythm, and generates audio waveforms that sound like a human voice.

Modern TTS systems like Kokoro use neural networks trained on thousands of hours of human speech. They learn not just how individual words sound but how sentences flow naturally, including pauses, emphasis, and intonation.

### What is Kokoro TTS?

Kokoro is an open-source neural TTS system that runs entirely on your local computer. Key facts:

- Completely free, no API keys, no usage limits
- Runs on CPU — no GPU required
- Multiple voice options (female, male, British, American)
- Produces natural-sounding speech, not the robotic TTS of older systems
- Uses ONNX format for the model, making it fast to run

We use the `kokoro-onnx` Python library which wraps the Kokoro model in an easy-to-use interface.

### What is ONNX?

ONNX stands for Open Neural Network Exchange. It is a standard format for storing trained machine learning models that can run on many different platforms and hardware configurations. The Kokoro model is stored as a `.onnx` file which the `kokoro-onnx` library loads and runs.

Think of ONNX like a PDF for AI models — just as a PDF can be opened on any device, an ONNX model can run on any compatible runtime.

### What is a .wav File?

WAV (Waveform Audio File Format) is an uncompressed audio format developed by Microsoft. Unlike MP3 which compresses audio by removing frequencies humans barely notice, WAV stores every sample of the audio signal. This makes WAV files:

- Larger than MP3 (about 10MB per minute of audio)
- Perfect quality with no compression artifacts
- Ideal for audio processing and mixing (Sprint 5 will mix these with music)

A 75-second spoken script produces roughly a 13MB WAV file.

### What is a Child Process?

Kokoro runs in Python but our app is built in Node.js (TypeScript). These are two completely separate programming runtimes that cannot directly call each other's code.

The solution is a child process — Node.js launches a separate Python process, sends it instructions via command-line arguments, and reads the results from its output.

```
Node.js (main process)
    |
    |-- spawn('python', ['tts.py', '--input', 'script.txt', '--output', 'audio.wav'])
    |
    Python (child process)
        |
        reads script.txt
        runs Kokoro
        writes audio.wav
        prints "SUCCESS: Audio saved..."
        exits
    |
Node.js reads the output
reads "SUCCESS" and knows it worked
reads "DURATION: 75.3" and knows how long the audio is
```

This pattern is called inter-process communication via stdio (standard input/output).

### What is Base64 Encoding?

When we play audio in the Electron renderer (the React UI), we cannot directly access files on the local file system. Electron's security model prevents the renderer from loading `file://` URLs for security reasons.

The solution: read the file in the main process (which has full file system access), convert the binary audio data to a Base64 string (text representation of binary data), send it to the renderer via IPC, then convert it back to a Blob URL that the audio player can use.

```
Main process:
  fs.readFileSync('audio.wav')     -- reads binary file
  .toString('base64')              -- converts to text like "UklGRiQ..."

IPC transfer (text travels safely)

Renderer process:
  atob(base64)                     -- converts text back to binary
  new Blob([...], {type:'audio/wav'})  -- creates browser-compatible object
  URL.createObjectURL(blob)        -- creates a playable URL like blob:// ...
  audioElement.src = blobUrl       -- audio player can now load it
```

### What is Audio Caching?

Generating a 75-second audio file takes real time (30-90 seconds depending on script length). We never want to generate the same audio twice.

The caching system works by checking if a file already exists before generating:

```
Request to generate audio for sub-segment ID 45
    |
    Does data/audio-output/segment_45.wav exist?
    |
    YES --> return the existing file path immediately
    NO  --> run Kokoro, generate the file, then return the path
```

This means clicking "Generate All Audio" after already generating one part only generates the remaining parts, not all of them again.

### What is a Voice File?

Kokoro separates the main model (which knows how to generate speech) from voice files (which define the specific characteristics of each voice). This is like separating a speech engine from a voice actor's recording.

Voice files we downloaded:

```
af_heart    -- warm American female voice (our default for Al)
af_bella    -- energetic American female
am_adam     -- American male
am_michael  -- deeper American male
bm_george   -- British male
bf_emma     -- British female
```

Each voice file is a `.pt` file (PyTorch tensor format) containing the acoustic characteristics of that voice.

## 5. File-by-File Walkthrough

### apps/desktop/src/python/tts.py

A standalone Python script that does one job: take a text file and produce a WAV file.

Why a standalone script instead of calling Python from Node directly? Because Python and Node.js are separate runtimes. Spawning a process is the standard way to bridge them. The script uses `argparse` to accept command-line arguments:

```
--input   path to the .txt file containing the script
--output  path where the .wav file should be saved
--voice   which Kokoro voice to use (default: af_heart)
--speed   speech speed multiplier (default: 1.0)
```

The script prints structured output that Node.js parses:

```
SUCCESS: Audio saved to /path/to/file.wav
DURATION: 75.3
```

Node.js reads these lines and extracts the duration using a regex:

```typescript
const durationMatch = stdout.match(/DURATION:\s*([\d.]+)/)
const durationSec   = durationMatch ? parseFloat(durationMatch[1]) : 0
```

### apps/desktop/src/main/voiceEngine.ts

The Node.js side of the TTS pipeline. Key functions:

**generateAudio(script, subSegmentId, voice, speed)**

1. Creates the output directory if it does not exist
2. Writes the script text to a temporary .txt file
3. Spawns the Python process with the correct arguments
4. Collects stdout and stderr as the process runs
5. On completion, parses the duration from stdout
6. Deletes the temporary .txt file
7. Returns success/failure with the audio path and duration

**generateAudioForSegments(subSegments, voice, onProgress)**

Loops through an array of sub-segments and calls `generateAudio` for each one. Accepts an `onProgress` callback so the main process can log progress like "TTS progress: 2/3".

**audioExists(subSegmentId)**

Checks if `data/audio-output/segment_{id}.wav` exists. Used for caching.

**getAudioPath(subSegmentId)**

Returns the expected file path for a given sub-segment ID. Used when serving cached audio.

### apps/desktop/src/main/main.ts additions

Three new IPC handlers:

**tts:generate**
Generates audio for a single sub-segment. Checks cache first, skips generation if file exists.

**tts:generateBatch**
Generates audio for multiple sub-segments. Filters out already-cached segments before processing, then merges cached and newly generated results.

**tts:getAudioData**
Reads a WAV file from disk and returns it as a Base64 string. Used by the renderer to play audio without file:// URL restrictions.

### apps/desktop/src/renderer/components/ScriptPanel.tsx additions

New state variables:

```typescript
audioPaths:  Record<number, string>   -- maps sub-segment ID to audio file path
audioStatus: Record<number, 'ready' | 'generating' | 'error'>  -- tracks generation state
audioRef:    RefObject<HTMLAudioElement>  -- reference to the <audio> DOM element
```

**playAudio(filePath)**

1. Calls `electronAPI.getAudioData(filePath)` to get Base64 audio
2. Converts Base64 to a Uint8Array
3. Creates a Blob with type `audio/wav`
4. Creates an object URL from the Blob
5. Sets the audio element's src and calls play()

**Generate Audio button**
Appears when a sub-segment has no audio yet. Calls `handleGenerateAudio` for the current tab.

**Generate All Audio button**
Generates audio for all sub-segments at once. Respects caching — only generates missing files.

**Musical note indicator**
A small green music note (♪) appears on tab buttons when audio has been generated for that part.

## 7. Problems We Hit and How We Fixed Them

### Kokoro model files moved repositories

The original model files at `hexgrad/Kokoro-82M` no longer contained `.onnx` files. The repository restructured to use PyTorch `.pth` format instead.

Fix: Downloaded from `thewh1teagle/kokoro-onnx` GitHub releases directly using the `requests` library, bypassing the Hugging Face hub entirely.

Lesson: Always verify model file locations before a sprint. Open source model repositories reorganize frequently.

### kokoro library required spacy which failed to compile

Installing the newer `kokoro` package pulled in `spacy` as a dependency, which requires C++ build tools to compile on Windows and failed.

Fix: Stayed with `kokoro-onnx` which was already installed and working. The ONNX version is simpler, has fewer dependencies, and works fine for our use case.

Lesson: Fewer dependencies is better. The simplest working solution beats the latest version.

### file:// URLs blocked in Electron renderer

Electron's security model prevents the renderer process from loading local files via `file://` URLs. Multiple approaches failed including custom protocol registration.

Fix: Read the file in the main process using Node.js `fs.readFileSync`, encode as Base64, send via IPC to the renderer, decode into a Blob URL. This approach works reliably because it uses Electron's own IPC channel rather than trying to bypass security restrictions.

Lesson: When Electron blocks a direct approach, route through the main process. The main process has full Node.js capabilities.

### Custom radio:// protocol stripping drive letter

Registering a custom `radio://` protocol lost the Windows drive letter colon (`C:` became `C`) when parsing the URL.

Fix: Abandoned the custom protocol entirely in favor of the Base64 IPC approach above.

Lesson: URL parsing on Windows paths is tricky. When fighting platform-specific path issues, change the approach rather than fighting the parsing.

### Double path in Python script location

`process.cwd()` in Electron returned `~/airadio/apps/desktop`, so joining it with `apps/desktop/src/python/tts.py` produced a doubled path.

Fix: Changed `PYTHON_SCRIPT` to use `path.join(process.cwd(), 'src', 'python', 'tts.py')` since `process.cwd()` already includes `apps/desktop`.

Lesson: Always log constructed paths during debugging. The doubled path was immediately obvious once logged.

## 9. Audio Cleanup Plan (Sprint 9)

Audio files accumulate over time. Each 75-second script produces roughly 13MB. Running the app daily for a month could produce gigabytes of audio files.

Planned cleanup function for Sprint 9:

```
On app startup:
  Scan data/audio-output/ for all .wav files
  Check file modification date
  Delete files older than 30 days
  Delete matching sub_segments records from SQLite
  Log: "Cleaned up X files, freed Y MB"
```

This keeps storage under control automatically without manual intervention.
